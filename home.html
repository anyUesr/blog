操作系统

  1. bashrc与环境变量

    1. bash的系统和个人配置文件的路径是什么?
      • 系统：
        • /etc/bashrc
        • /etc/profile
        • /etc/bash_logout
      • 个人：
        • $HOME/.bashrc
        • $HOME/.bash_logout
        • $HOME/._profile
    2. $PS1这个环境变量有什么用?
      • 设置提示符，可根据需要显示当前时间、所在文件夹等信息

    3. 在什么情况下需要修改$PATH, 应该如何合理地修改它?

      • 在需要不使用绝对路径来执行可执行文件时，要将可执行文件所在文件夹加入PATH

      • 引用自 /etc/profile

        PATH=”/usr/local/bin:/usr/bin:/bin:${PATH}”
        export PATH

        可根据优先顺序将对应文件夹置入PATH，越靠前则优先级越高

    4. 修改过~/.bashrc后, 如何让改变立即生效?
      • . ~/.bashrc
    5. bashrc与profile有什么异同点? 两者的加载顺序如何?
      • 异：
        • profile 仅在第一次登陆时执行，通过 [[ -f ~/.bashrc ]] && . ~/.bashrc 加载bashrc，一般在 bashprofile 修改PATH 及其他环境变量
        • bashrc 会在每次开启新终端时执行，
      • 同：
        • 一般在 bashrc 中编写交互式的脚本，将登陆后的默认操作写入bashrc中
      • 执行顺序：先 profile 后 bashrc

      • 系统自带
        • /bin ：普通用户和root都可以使用
        • /sbin ：仅root可以使用
      • 额外安装：
        • /usr/bin 与 /usr/sbin ：差别同 /bin 与 /sbin
    2. /var目录通常用来放哪些内容? /var和/tmp有什么区别?
      • /var var 即 variable，动态内容，用来存放经常变动但需要长期保存的信息，比如 mysql 的默认存储目录， apache 的静态页面默认存储目录就在 /var
      • /var 存储的数据需要长期保存，/tmp 只是缓存，无需长期保存

  4. NTP与UTC

    1. ntpdate和ntpd相比有什么优势和劣势.
      • ntpd在实际同步时间是逐步校准时间
        • 优点：对业务影响小
        • 缺点：见效慢
      • ntpdate不会考虑其他因素，直接调整时间
        • 优点：简单快速，当时见效，适合系统初装，还未上线使用或刚刚重新启动后
        • 缺点：已上线的机器不能用，否则会导致某些数据（如数据库中的时间数据错乱）
    2. 既然已经有了ntp, 为什么还需要有hwclock?
      • 避免重启后丢失
    3. UTC是什么?
      • 时间协调时间，会尽量接近格林尼治标准时间
    4. Debian中如何修改系统时区?
      • 使用 tzselect

  5. 硬链接与软链接

    假设 B 是 A 的(软/硬)链接文件, 回答下列问题
    1. 对于硬链接和软链接, 比较当A或者B被删除时, 各有什么后果.
      • hard：A B 删除任意一个，对另一个无影响，文件内容不变
      • soft：
        • B 被删除，对 A 无影响，可继续对A直接进行操作；
        • A 被删除，则无法通过 B 来操作A
    2. 比较创建硬链接和软链接, 各需要多少硬盘空间?
      • hard：不额外占用空间
      • soft：会额外占用数个字节的空间
    3. 为什么不能对目录建立硬链接? 为什么不能跨设备建立硬链接?
      • 不能对目录建立硬链接，主要是为逻辑需求，如果对当前目录的父目录简历硬链接，则 ‘..’ 就无法甄别出该显示哪个目录
      • 硬链接是通过修改 inode 实现的，inode 不能跨设备，硬链接也不能跨设备
    4. 当A不存在时, 能否建立A的硬链接B? 能否建立A的软链接B?
      • hard：可以
      • soft：不可以

  7. Linux系统启动

    1. 请参考各种文档, 综述Linux系统启动过程中发生了什么事情.
      • bios读取mbr，启动grub，grub加载内核，进行系统初始化，创建内存环境，加载硬盘文件系统等等，启动系统服务，开启login程序
    2. Debian系统服务启动的顺序是如何被确定的?
      • 通过/etc/rcx.d/下SXX文件名来决定启动顺序，XX越大越靠后
    3. 运行级别是什么东西?
      • 启动的模式
        • 1：单用户模式
        • 2：无网络多用户模式
        • 3：有网络多用户模式
        • 5：图形模式多用户模式
    4. 单用户模式是什么东西?
      • 在此模式下，只加载必要驱动和服务，无网络支持，不支持多用户，一般在服务故障无法开机或忘记root密码时使用

  8. 管道与重定向

    1. 重定向中的0, 1, 2各代表什么意思?
      • 0：标准输入

    2. 请简述管道是如何让程序能够互相协同工作的.

    3. 命名管道是什么东西? 如何创建一个有名管道?

      • 命名管道即FIFO管道文件，将字符串读出，则字符串在管道文件中不复存在

      • 使用 mkfifo 命令即可创建

          mkfifo -m 600 /tmp/test.fifo
    4. 请简述一个场景, 其中使用命名管道是必要的.
      • shell脚本进行多进程并发，使用命名管道进行并发数量控制

    3. apt-get install 是如何处理已安装但需要升级的软件(例如openssh)的升级的?
      • 如果软件被设置为 hold 则不升级，没有被设置为 hold 则可以升级

  11. 安全习惯

    1. 请从攻击者的角度来讲述有多少种方式来获得公用wifi中的其他用户的信息.
      • arp 欺骗
    2. 请解释公私钥加密的原理, 并解释应该如何保管和使用ssh私钥.
      1. cilent 向 server 请求连接
      2. server 使用 client 对应用户目录下的 .ssh/authorized_keys 中对应的公钥加密一段随机字符串发送给 client，client 将字符串用相应私钥解密，并将解密结果发送给 server，server 检查解密后字符串与加密前是否相等，相等则验证通过
    3. 请解释ssh agent forward的工作原理, 并从攻击者的角度来讲述这一工作原理有没有可利用之处.
      • ssh agent forward 意为验证代理，即通过跳转机 A 登陆 跳转机 B 时，无需输入密码，具体配置是将同一公钥发送至 A 和 B，并将 A 的 sshd_config 的 ForwardAgent 设置为 yes。
      • 原理为 A 将 B 加密后的字符串转发送至本机，并将本机解密后的字符串转发至 B 完成验证
      • 风险：如果 root 也使用了 ssh agent forward ，则低权限用户通过修改 $SSH_AUTH_SOCK 即可伪装成 root 登陆至其他服务器
    4. 列举你所知道的密码管理工具及其工作方式, 并从攻击者角度来讲述这些工具有什么安全隐患.
      • 只知道 keypass ，将密码进行本地加密并存储，用时解密并粘贴至相应输入框。
      • 隐患：如果电脑被别人使用，即可直接使用密码进行相应登陆。
    5. https全站加密有什么优势和劣势?
      • 优势：保密性好，简化验证过程，减小开发压力，抓包破解风险减小
      • 劣势：对服务器压力过大，可能需要 ssl 加密卡
    6. 请阐述你对最小权限原则的理解.
      • 能不给的权限就不给
      • 给的权限要及时收回

  13. 信号机制

    1. 请简述kill命令的原理.
      • 对进程发送相应信号，如果进程自己实现了对相应信号的处理方式，会以相应方式进行处理；如果没有，内核通过会按照默认的方式处理进程
    2. 请简述nohup命令的原理.
      • 进程属主在 exit 后，终端会收到 SIGHUP信号从而关闭属主的所有子进程，但进程可以自定义对 SIGHUP 信号的处理方法，nohup 即是让进程无视 SIGHUP 继续运行

    3. 如何在bash脚本中处理用户的 Ctrl-C ?
      通过 trap 捕捉 相应信号并处理

       #!/bin/bash
 trap "echo 'Ctrl - C'" INT
 trap "echo 'Ctrl - \';exit 0" QUIT
 while :
 do
     echo "running"
     sleep 2
 done

    5. top里的load average是如何计算的?
      • 如满足以下几点
        • 它没有在等待I/O操作的结果
        • 它没有主动进入等待状态(也就是没有调用’wait’)
        • 没有被停止(例如：等待终止)

    6. 假设top显示ffmpeg进程的CPU使用率为143.7%, 请具体解释这个数值是如何计算出来的.

      • 超过 100% 说明进程一定使用了多个核心

  2. ssh

    1. 请简要叙述ssh连接加密的原理

    2. 请简述ssh agent forward的原理
      • 原理为跳转机将远程 server 公钥加密后的字符串转发送至本机，并将本机私钥解密后的字符串转发至远程 server 完成验证
    3. ssh连接时如何指定远程端口, 如何设置连接超时时间?

    4. ssh私钥文件默认的权限是什么?

      • 示例：使用 HOST2 转发HOST1数据至 HOST3
      • 本地转发： ssh -L 10240:HOST3:22 HOST2 - HOST1执行

      IPv4 报文格式

      • 版本
        IP报文首部的第一个字段是4位版本字段。对IPv4来说，这个字段的值是4。
      • 首部长度（IHL）
        第二个字段是4位首部长度，说明首部有多少32位字长。由于IPv4首部可能包含数目不定的选项，这个字段也用来确定数据的偏移量。这个字段的最小值是5（RFC 791），最大值是15。
      • DiffServ（DSCP）
        最初被定义为服务类型字段，但被RFC 2474重定义为DiffServ。新的需要实时数据流的技术会应用这个字段，一个例子是VoIP。
      • 显式拥塞通告（ECN）
        在RFC 3168中定义，允许在不丢弃报文的同时通知对方网络拥塞的发生。ECN是一种可选的功能，仅当两端都支持并希望使用，且底层网络支持时才被使用。
      • 全长
        这个16位字段定义了报文总长，包含首部和数据，单位为字节。这个字段的最小值是20（20字节首部+0字节数据），最大值是65,535。所有主机都必须支持最小576字节的报文，但大多数现代主机支持更大的报文。有时候子网会限制报文的大小，这时报文就必须被分片。
      • 标识符
        这个字段主要被用来唯一地标识一个报文的所有分片。一些实验性的工作建议将此字段用于其它目的，例如增加报文跟踪信息以协助探测伪造的源地址。[5]

      • 标志
        这个3位字段用于控制和识别分片，它们是：

        • 位0：保留，必须为0；
        • 位1：禁止分片（DF）；
        • 位2：更多分片（MF）。

        如果DF标志被设置但路由要求必须分片报文，此报文会被丢弃。这个标志可被用于发往没有能力组装分片的主机。
        当一个报文被分片，除了最后一片外的所有分片都设置MF标志。不被分片的报文不设置MF标志：它是它自己的最后一片。

      • 分片偏移
        这个13位字段指明了每个分片相对于原始报文开头的偏移量，以8字节作单位。
      • 存活时间（TTL）
        这个8位字段避免报文在互联网中永远存在（例如陷入路由环路）。存活时间以秒为单位，但小于一秒的时间均向上取整到一秒。在现实中，这实际上成了一个跳数计数器：报文经过的每个路由器都将此字段减一，当此字段等于0时，报文不再向下一跳传送并被丢弃。常规地，一份ICMP报文被发回报文发送端说明其发送的报文已被丢弃。这也是traceroute的核心原理。
      • 协议
        这个字段定义了该报文数据区使用的协议。IANA维护着一份协议列表（最初由RFC 790定义）。
      • 首部检验和
        这个16位检验和字段用于对首部查错。在每一跳，计算出的首部检验和必须与此字段进行比对，如果不一致，此报文被丢弃。值得注意的是，数据区的错误留待上层协议处理----用户数据报协议和传输控制协议都有检验和字段。
        因为生存时间字段在每一跳都会发生变化，意味着检验和必须被重新计算，RFC 1071这样定义计算检验和的方法：
      • 源地址
        一个IPv4地址由四个字节共32位构成，此字段的值是将每个字节转为二进制并拼在一起所得到的32位值。
        例如，10.9.8.7是00001010000010010000100000000111。
        这个地址是报文的发送端。但请注意，因为NAT的存在，这个地址并不总是报文的真实发送端，因此发往此地址的报文会被送往NAT设备，并由它被翻译为真实的地址。
      • 目的地址
        与源地址格式相同，但指出报文的接收端。
      • 选项
        附加的首部字段可能跟在目的地址之后，但这并不被经常使用。请注意首部长度字段必须包括足够的32位字来放下所有的选项（包括任何必须的填充以使首部长度能够被32位整除）。当选项列表的结尾不是首部的结尾时，EOL（选项列表结束，0x00）选项被插入列表末尾。下表列出了可能的选项：

        数字 5 指明一个选项

        数据字段不是首部的一部分，因此并不被包含在检验和中。数据的格式在协议首部字段中被指明，并可以是任意的传输层协议。

    7. DHCP 协议的原理和 DHCP Client 获取地址的基本过程

      1. 由于客户机最初不知道DHCP服务器的IP地址，因此设置使用DHCP的客户端将以广播方式向网络中发送DHCP discover信息，用来查找网络中是否存在DHCP服务器。这样，网络中每一台计算机都可收到该广播信息，但只有DHCP服务器才会做出响应。

      2. DHCP服务器收到客户端发送的DHCP discover信息，将从DHCP服务器中还没有分配的IP地址中选择一个，并将IP地址、子网掩码、网关地址等信息发送给申请的客户端计算机（DHCP offer信息）。

      3. 由于网络中可能存在多台DHCP服务器，而这些DHCP服务器都可能会向申请IP地址的客户端返回一个IP地址及相关信息。这时，就需要设置一种选择方案，通常的规则是DHCP客户端接受第1个收到的分配IP地址信息（DHCP offer信息）。

      4. 客户端以广播方式发出一个DHCP request信息，该信息中包含向它所选定的DHCP服务器请求IP地址的内容。这里使用广播方式来应答，是为了通知所有其他DHCP服务器，该客户端已选择了某台DHCP服务器所提供的IP地址。

      5. 当DHCP服务器收到DHCP客户端广播的DHCP request信息之后，被使用的DHCP服务器向DHCP客户端发送一个包含其所提供的IP地址和其他设置的DHCP ACK信息，告诉DHCP客户端可以使用该IP地址，其他的DHCP服务器都将收回曾提供的IP地址。而DHCP客户端便将收到的IP地址与客户端的网卡绑定

    8. 路由表的组成，路由匹配过程，最长匹配原则
      对每组网络界面(interface)，路由表至少会存有下面的资讯：

      • 网络ID（Network ID, Network number）：就是目标地址的网络ID。
      • 子网掩码（subnet mask）：用来判断 IP 所属网络
      • 下一跳地址/接口（Next hop / interface）：就是数据在发送到目标地址的旅途中下一站的地址。

      按子网掩码长度排列，子网掩码长度越长越优先匹配

  2. NAT原理与应用

    1. NAT的基本原理
      • 通过 NAT 表进行地址转换
      • 当信息由本地网络向因特网传递时，源地址被立即从专有地址转换为公用地址
      • 由路由器跟踪每个连接上的基本数据，主要是目的地址和端口。当有回复返回路由器时，它通过输出阶段记录的连接跟踪数据来决定该转发给内部网的哪个主机
    2. Linux支持的NAT类型
      • SNAT
      • DNAT
    3. NAT对TCP、UDP、SSL、HTTP等各种应用和协议的影响
      • 把端口转发、LVS 和集群同步做好了的话，好像也没什么影响
    4. Linux中如何配置NAT
      • iptables -t nat -A POSTROUTING -j SNAT --to-source 192.168.0.1
        SNAT 一般不更改源端口，否则容易引发 TIME-WAIT 问题
      • iptables -t nat -A PREROUTING -p tcp -d 202.112.1.3 --dport 80 -j DNAT --to-destination 192.168.0.2:8080

        Usage: traceroute [ -46dFITnreAUDV ] [ -f first_ttl ] [ -g gate,… ] [ -i device ] [ -m max_ttl ] [ -N squeries ] [ -p port ] [ -t tos ] [ -l flow_label ] [ -w waittime ] [ -q nqueries ] [ -s src_addr ] [ -z sendwait ] [ --fwmark=num ] host [ packetlen ]

        Options

        option info
        -4 Use IPv4
        -6 Use IPv6
        -d --debug Enable socket level debugging
        -F --dont-fragment Do not fragment packets
        -f first_ttl --first=first_ttl Start from the first_ttl hop (instead from 1)
        -g gate,… --gateway=gate,… Route packets through the specified gateway (maximum 8 for IPv4 and 127 for IPv6)
        -I --icmp Use ICMP ECHO for tracerouting
        -T --tcp Use TCP SYN for tracerouting (default port is 80)
        -i device --interface=device Specify a network interface to operate with
        -m max_ttl --max-hops=max_ttl Set the max number of hops (max TTL to be reached). Default is 30
        -N squeries --sim-queries=squeries Set the number of probes to be tried simultaneously (default is 16)
        -n Do not resolve IP addresses to their domain names
        -p port --port=port Set the destination port to use. It is either initial udp port value for “default” method (incremented by each probe, default is 33434), or initial seq for “icmp” (incremented as well, default from 1), or some constant destination port for other methods (with default of 80 for “tcp”, 53 for “udp”, etc.)
        -t tos --tos=tos Set the TOS (IPv4 type of service) or TC (IPv6 traffic class) value for outgoing packets
        -l flow_label --flowlabel=flow_label Use specified flow_label for IPv6 packets
        -w waittime --wait=waittime Set the number of seconds to wait for response to a probe (default is 5.0). Non-integer (float point) values allowed too
        -q nqueries --queries=nqueries Set the number of probes per each hop. Default is 3
        -r Bypass the normal routing and send directly to a host on an attached network
        -s src_addr --source=src_addr Use source src_addr for outgoing packets
        -z sendwait --sendwait=sendwait Minimal time interval between probes (default 0). If the value is more than 10, then it specifies a number in milliseconds, else it is a number of seconds (float point values allowed too)
        -e --extensions Show ICMP extensions (if present), including MPLS
        -A --as-path-lookups Perform AS path lookups in routing registries and print results directly after the corresponding addresses
        -M name --module=name Use specified module (either builtin or external) for traceroute operations. Most methods havetheir shortcuts (-I’ means -M icmp’ etc.)
        -O OPTS,… --options=OPTS,… Use module-specific option OPTS for the traceroute module. Several OPTS allowed,separated by comma. If OPTS is “help”, print info about available options
        --sport=num Use source port num for outgoing packets. Implies ‘-N 1’
        --fwmark=num Set firewall mark for outgoing packets
        -U --udp Use UDP to particular port for tracerouting(instead of increasing the port per each probe),default port is 53
        -UL Use UDPLITE for tracerouting (default dest port is 53)
        -D --dccp Use DCCP Request for tracerouting (default port is 33434)
        -P prot --protocol=prot Use raw packet of protocol prot for tracerouting Discover MTU along the path being traced. Implies ‘-F -N 1’
        --back Guess the number of hops in the backward path and print if it differs
        -V --version Print version info and exit
        --help Read this help and exit

        Arguments

        option info
        host The host to traceroute to

    3. mtr命令的原理、用法等

      • 用法： mtr [-hvrctglspeniuTP46] [--help] [--version] [--report] [--report-wide] [--report-cycles COUNT][--curses] [--split] [--raw] [--mpls] [--no-dns] [--show-ips] [--gtk] [--address IP.ADD.RE.SS][--interval SECONDS] [--psize BYTES | -s BYTES] [--tcp] [--port PORT] [--timeout SECONDS] HOSTNAME [PACKETSIZE]

        OPTIONS

        option info
        -h --help Print the summary of command line argument options.
        -v --version Print the installed version of mtr.
        -r --report This option puts mtr into report mode. When in this mode, mtr will run for the number of cycles specified by the -c option, and then print statistics and exit.

        This mode is useful for generating statistics about network quality. Note that each running instance of mtr generates a significant amount of network traffic.Using mtr to measure the quality of your network may result in decreased network performance.
        -w --report-wide This option puts mtr into wide report mode. When in this mode, mtr will not cut hostnames inthe report.
        -c COUNT
        --report-cycles COUNT Use this option to set the number of pings sent to determine both the machines on the network and the reliability of those machines. Each cycle lasts one second.
        -s BYTES
        --psize BYTES
        PACKETSIZE These options or a trailing PACKETSIZE on the command line sets the packet size used for prob‐ing. It is in bytes inclusive IP and ICMP headers
        If set to a negative number, every iteration will use a different, random packet size upto that number.
        -t --curses Use this option to force mtr to use the curses based terminal interface (if available).
        -e --mpls Use this option to tell mtr to display information from ICMP extensions for MPLS (RFC 4950)that are encoded in the response packets
        -n --no-dns Use this option to force mtr to display numeric IP numbers and not try to resolve the host names.
        -b --show-ips Use this option to tell mtr to display both the host names and numeric IP numbers. In split mode this adds an extra field to the output. In report mode, there is usually too little space to add the IPs, and they will be truncated. Use the wide report (-w) mode to see the IPs in report mode.
        -o --order fields order Use this option to specify the fields and their order when loading mtr.

        Example: -o “LSD NBAW”
        -p --split Use this option to set mtr to spit out a format that is suitable for a split-user interface.
        -l --raw Use this option to tell mtr to use the raw output format.This format is better suited for archival of the measurement results. It could be parsed to be presented into any of the other display methods.
        -a --address IP.ADD.RE.SS Use this option to bind outgoing packets’ socket to specific interface, so that any packet will be sent through this interface. NOTE that this option doesn’t apply to DNS requests(which could be and could not be what you want).
        -i SECONDS
        --interval SECONDS Use this option to specify the positive number of seconds between ICMP ECHO requests.The default value for this parameter is one second.
        -u Use UDP datagrams instead of ICMP ECHO.
        -T --tcp Use TCP SYN packets instead of ICMP ECHO. PACKETSIZE is ignored, since SYN packets can not contain data.
        -P --port PORT The target port number for TCP traces.
        --timeout SECONDS The number of seconds to keep the TCP socket open before giving up on the connection. This will only affect the final hop. Using large values for this, especially combined with a short interval, will use up a lot of file descriptors.

        B Min/Best RTT(ms)
        A Average RTT(ms)
        W Max/Worst RTT(ms)
        V Standard Deviation
        G Geometric Mean
        J Current Jitter
        M Jitter Mean/Avg.
